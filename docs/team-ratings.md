# Team Ratings Explained

This document explains how the team ratings in [team-ratings.json](/Users/chad/source/tecmogeek/data/team-ratings.json) are generated by [rebuild-team-ratings.ts](/Users/chad/source/tecmogeek/scripts/rebuild-team-ratings.ts).

It is written for someone seeing the system for the first time.

## What this script does

The script reads each team roster file in `/data`, such as [49ers.json](/Users/chad/source/tecmogeek/data/49ers.json), and produces one derived record per team in [team-ratings.json](/Users/chad/source/tecmogeek/data/team-ratings.json).

It does not calculate player ratings from raw Tecmo attributes. It assumes those player ratings already exist in the team files.

The command is:

```sh
pnpm rebuild:team-ratings
```

## What data it uses

The generator uses the ratings already stored on players in the team JSON files:

- QBs use `rating`
- Offensive skill players use `rushing_rating` and `receiving_rating`
- Kick and punt returners use `kick_return_rating` and `punt_return_rating`
- Offensive linemen, defensive linemen, linebackers, and defensive backs use `rating`

The script converts percent strings like `"77%"` into numbers like `77`.

## High-level output

Each team gets three top-level ratings:

- `offensive_rating`
- `defensive_rating`
- `overall_rating`

The overall rating is always:

```text
overall_rating = 0.50 * offensive_rating + 0.50 * defensive_rating
```

The output also includes:

- offense and defense component scores
- the selected QB
- the selected running, passing, and balanced skill lineups
- the top kick returner and punt returner

## Team file discovery

The script reads every `.json` file in `/data` except the aggregate output files:

- `qb.json`
- `rushers.json`
- `receivers.json`
- `kick-returners.json`
- `punt-returners.json`
- `ol.json`
- `dl.json`
- `lb.json`
- `cb-s.json`
- `k.json`
- `p.json`
- `team-ratings.json`

The remaining files are treated as team roster files and are processed in sorted order by team slug.

## Core Tecmo modeling assumptions

The team rating model uses these gameplay assumptions:

- Offensive skill players are treated as one interchangeable pool: RB, WR, and TE players can be reassigned to offensive skill slots.
- Offensive slots are static once chosen for a lineup. The model does not assume the user is constantly changing player positions between plays.
- The five offensive skill slots are:
  - `RB1`
  - `RB2`
  - `WR1`
  - `WR2`
  - `TE1`
- Defense is treated as fixed starters only:
  - `RE`
  - `NT`
  - `LE`
  - `ROLB`
  - `RILB`
  - `LILB`
  - `LOLB`
  - `RCB`
  - `LCB`
  - `FS`
  - `SS`

## Quarterback selection

The script finds the two quarterbacks by position prefix `QB`.

It sorts them by QB `rating` and labels them:

- `best_qb`
- `backup_qb`

There is also an internal quarterback room score:

```text
qb_room = 0.80 * best_qb.rating + 0.20 * backup_qb.rating
```

The offensive formulas use `qb_room`, not just the best QB's rating.

The best QB's individual score is still recorded as `qb_rating` for reference:

```text
qb_rating = best_qb.rating
```

So:

- `qb_room` is used in the offense formulas
- `qb_rating` is included only as context
- `best_qb` and `backup_qb` are both recorded in the output

## Returners

The script separately finds the best kick returner and punt returner from the RB/WR/TE pool.

Those scores are:

```text
kick_return_score = 0.75 * best_kr + 0.25 * second_kr
punt_return_score = 0.75 * best_pr + 0.25 * second_pr
return_room_score = 0.55 * kick_return_score + 0.45 * punt_return_score
```

The offensive formulas use `return_room_score`.

The output includes only:

- `kr`
- `pr`

The output also includes `return_room` in the `components` object.

## Offensive line score

The offensive line score is a simple average of:

- `C`
- `LG`
- `RG`
- `LT`
- `RT`

Formula:

```text
ol_line = average(C, LG, RG, LT, RT)
```

## How skill lineups are built

For offense, the script looks at exactly 10 skill players:

- all RBs
- all WRs
- all TEs

It requires exactly 10 skill players in the roster file.

For each lineup type, it searches every 5-player combination from those 10 players.

That means it evaluates:

```text
10 choose 5 = 252
```

candidate groups for each lineup type.

Then it assigns those 5 players to the five Tecmo offensive slots.

## Running lineup

The running lineup is intended to model the strongest run-oriented static offense.

### Slot assignment

The running lineup is assigned in this order:

1. Choose `TE1` as a support role, not a runner.
2. Among the remaining four players, put the two best rushers in `RB1` and `RB2`.
3. Put the final two remaining players in `WR1` and `WR2`, ordered by rushing rating.

### Why `TE1` is special in the running lineup

The model assumes `TE1` is not a true ballcarrier slot in the running game.

So `TE1` is chosen as a support slot using this rule:

- Prefer a player with `hitting_power >= 75`
- If there is no such player, use the best available receiver

The support score used for running `TE1` is:

```text
if hitting_power >= 75:
  running_te_support_score = 95 + 0.05 * receiving_rating
else:
  running_te_support_score = receiving_rating
```

This intentionally gives strong priority to high-hitting-power support players when selecting `TE1` for the running lineup.

### Running skill score

Once the slots are assigned, the running skill score is:

```text
running_skill =
  0.40 * RB1.rushing_rating +
  0.30 * RB2.rushing_rating +
  0.12 * WR1.rushing_rating +
  0.10 * WR2.rushing_rating +
  0.08 * TE1.running_te_support_score
```

## Passing lineup

The passing lineup is intended to model the strongest pass-oriented static offense.

### Slot assignment

The passing lineup is assigned in this order:

1. Put the three best receivers into `WR1`, `WR2`, and `TE1`
2. Put the two remaining players into `RB1` and `RB2`, ordered by rushing rating

The three receiver slots are ordered by `receiving_rating`, highest to lowest.

### Passing skill score

Once the slots are assigned, the passing skill score is:

```text
passing_skill =
  0.10 * RB1.receiving_rating +
  0.15 * RB2.receiving_rating +
  0.30 * WR1.receiving_rating +
  0.25 * WR2.receiving_rating +
  0.20 * TE1.receiving_rating
```

## Balanced lineup

The balanced lineup is intentionally pass-biased.

It is based on the idea that in Tecmo, passing is generally the more effective offensive mode, so the balanced offense should still prioritize strong receiving slots.

### Slot assignment

The balanced lineup uses the same slot assignment rule as the passing lineup:

1. Put the three best receivers into `WR1`, `WR2`, and `TE1`
2. Put the two remaining players into `RB1` and `RB2`

### Balanced skill score

Even though the slots are assigned pass-first, the score still includes both running and passing value.

The current balanced skill score is:

```text
balanced_skill =
  0.25 * running_skill +
  0.75 * passing_skill
```

This means the balanced lineup has a clear passing bias both in slot assignment and in scoring.

## How the best skill lineup is chosen

For each lineup type:

- running
- passing
- balanced

the script:

1. enumerates every 5-player combination from the 10 skill players
2. assigns those 5 players to the five slots using that lineup type's rules
3. calculates the lineup score
4. keeps the highest-scoring result

If two lineups have exactly the same score within a very small floating-point tolerance, the script breaks the tie by comparing a deterministic lineup signature string in alphabetical order.

That makes the result stable across runs.

## Offensive rating

The final offense is built from three offensive sub-ratings:

- `running_offense`
- `passing_offense`
- `balanced_offense`

### Running offense

```text
running_offense =
  0.63 * running_skill +
  0.27 * ol_line +
  0.10 * return_room
```

### Passing offense

```text
passing_offense =
  0.45 * qb_room +
  0.315 * passing_skill +
  0.135 * ol_line +
  0.10 * return_room
```

### Balanced offense

```text
balanced_offense =
  0.27 * qb_room +
  0.18 * ol_line +
  0.45 * balanced_skill +
  0.10 * return_room
```

### Final offensive rating

```text
offensive_rating =
  0.25 * running_offense +
  0.25 * passing_offense +
  0.50 * balanced_offense
```

So the final offense gives the most weight to the balanced offense.

## Defensive rating

Defense uses only the fixed 11 starters.

There is no defensive substitution modeling.

### Defensive groups

The defense is split into three groups:

- defensive line: `RE`, `NT`, `LE`
- linebackers: `ROLB`, `RILB`, `LILB`, `LOLB`
- secondary: `RCB`, `LCB`, `FS`, `SS`

Each group score is just the average of the players in that group:

```text
dl_group = average(RE, NT, LE)
lb_group = average(ROLB, RILB, LILB, LOLB)
secondary_group = average(RCB, LCB, FS, SS)
```

### Final defensive rating

The defensive rating is weighted by the number of fixed starters in each unit:

```text
defensive_rating =
  (3 * dl_group + 4 * lb_group + 4 * secondary_group) / 11
```

This means the defense is effectively weighted by:

- 3 defensive linemen
- 4 linebackers
- 4 defensive backs

## Final overall rating

The overall team rating is:

```text
overall_rating = 0.50 * offensive_rating + 0.50 * defensive_rating
```

No extra team-level bonus, durability adjustment, or special-teams adjustment is added after that.


## Important things this system does not currently do

This is just as important as what it does do.

The current team rating system does **not** currently:

- use kicker or punter ratings
- model injuries or bench depth beyond the chosen static lineups
- model defensive substitutions
- change player positions dynamically between plays
- read playbooks or down-and-distance tendencies

So this is best understood as a static lineup strength model derived from the player ratings already stored in the repo.

## Short summary

If you want the shortest accurate summary, it is this:

1. Read the player ratings already stored in each team file.
2. Build three best-possible five-man offensive skill lineups:
   - running
   - passing
   - balanced, with a passing bias
3. Combine those with the best QB and the offensive line to get `offensive_rating`.
4. Average the fixed defensive starters by unit size to get `defensive_rating`.
5. Average offense and defense 50/50 to get `overall_rating`.
